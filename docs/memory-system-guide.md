# 永久记忆系统使用指南

## 功能概述

永久记忆系统为OPRCLI提供了长期记忆能力，包括：
- ✅ **基础CRUD**：创建、读取、更新、删除记忆
- ✅ **智能搜索**：按关键词、标签、时间范围搜索
- ✅ **分层总结**：5层时间金字塔，自动压缩历史信息
- ✅ **Load命令**：一键触发记忆总结和整合

## 核心架构

### 双层存储结构

```
┌─────────────────────────────────────┐
│      用户对话（自动记录）              │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│    原始记忆层 (Raw Memories)          │
│  - 永久保存所有对话记录               │
│  - 支持搜索和标签                     │
└──────────────┬──────────────────────┘
               │ Load 命令
               ▼
┌─────────────────────────────────────┐
│    总结记忆层 (Summaries)             │
│  Level 0: 1小时范围（详细）           │
│  Level 1: 1天范围（较详细）           │
│  Level 2: 1周范围（中等）             │
│  Level 3: 1月范围（简略）             │
│  Level 4: 1季度范围（概要）           │
└─────────────────────────────────────┘
```

## 使用方法

### 1. 自动记忆记录

系统会自动记录所有对话：
- 用户消息 → 标签：`user`, `message`
- AI回复 → 标签：`assistant`, `response`

### 2. 命令使用

#### 🔄 Load 命令（核心功能）

```
load
```

**功能**：
1. 总结所有未总结的记忆
2. 整合已总结的记忆（从低层到高层）
3. 生成时间金字塔结构

**示例输出**：
```
📊 记忆总结完成

📝 原始记忆：156 条
✨ 新总结：3 条
🔗 合并总结：2 条

📈 当前总结层级：
  Level 0 (1小时): 0 条
  Level 1 (1天): 5 条
  Level 2 (1周): 2 条
  Level 3 (1月): 1 条
  Level 4 (1季度): 0 条
```

#### 🔍 搜索记忆

```
search 钉钉
搜索 钉钉
```

**功能**：
- 按关键词搜索记忆内容
- 支持中英文搜索
- 默认返回前10条结果

**示例输出**：
```
🔍 搜索结果 (3 条)

1. 2026/3/4 10:30:15 [dingtalk, robot]
   用户询问了关于钉钉机器人的问题...

2. 2026/3/3 15:20:00 [dingtalk, setup]
   钉钉机器人设置已完成...

3. 2026/3/2 09:15:30 [dingtalk, error]
   钉钉连接出现错误...
```

#### 📊 统计信息

```
stats
统计
```

**功能**：
- 查看记忆总数
- 查看未总结/已总结数量
- 查看各层级总结数量
- 查看存储占用

**示例输出**：
```
📊 记忆系统统计

📝 原始记忆：
  总计: 156 条
  未总结: 12 条
  已总结: 144 条

📑 总结层级：
  Level 0 (1小时): 0 条
  Level 1 (1天): 5 条
  Level 2 (1周): 2 条
  Level 3 (1月): 1 条
  Level 4 (1季度): 0 条

💾 存储：
  原始记忆: 45.32 KB
  总结: 12.56 KB
```

#### 🏷️ 添加标签

```
tag abc123 重要 项目
```

**功能**：
- 为指定记忆添加标签
- 自动去重
- 支持多个标签

## 工作流程

### 正常使用流程

1. **日常对话** → 系统自动记录到原始记忆层
2. **定期触发** → 用户输入 `load` 命令
3. **自动总结** → 系统生成 Level 0 总结（1小时范围）
4. **整合升级** → 系统将低层总结合并为高层总结
5. **压缩存储** → 旧记忆被压缩为概要，节省空间

### 时间金字塔示例

```
最新（Level 0 - 1小时）
  ├─ 10:00-10:30: 详细记录每条对话
  ├─ 10:30-11:00: 详细记录每条对话
  └─ 11:00-11:30: 详细记录每条对话
         ↓ 合并
最近（Level 1 - 1天）
  ├─ 今天上午: 概要总结（包含3个Level 0）
  ├─ 今天下午: 概要总结（包含3个Level 0）
  └─ 今天晚上: 概要总结（包含3个Level 0）
         ↓ 合并
短期（Level 2 - 1周）
  ├─ 本周一: 概要总结（包含多个Level 1）
  ├─ 本周二: 概要总结（包含多个Level 1）
  └─ ...
         ↓ 合并
中期（Level 3 - 1月）
  ├─ 第一周: 重要事件摘要
  ├─ 第二周: 重要事件摘要
  └─ ...
         ↓ 合并
长期（Level 4 - 1季度）
  ├─ 一月: 概要统计
  ├─ 二月: 概要统计
  └─ 三月: 概要统计
```

## 最佳实践

### 1. 定期执行 Load

建议每天执行1-2次 `load` 命令：
- 早上：总结昨天的对话
- 晚上：总结今天的对话

### 2. 合理使用标签

重要对话可以手动添加标签：
```
tag abc123 重要 需要跟进
```

### 3. 定期清理

虽然原始记忆永久保存，但可以查看统计信息：
```
stats
```

如果总结过多，可以：
- 删除旧的总结文件
- 重新执行 `load` 重建

## 技术细节

### 性能优化

- **增量总结**：只总结未总结的记忆
- **时间窗口分组**：避免单次总结过多记录
- **软删除**：删除操作只标记，实际数据保留

### 存储优化

- **JSON格式**：人类可读，易于调试
- **压缩存储**：高层总结占用空间更小
- **索引优化**：按时间排序，快速检索

### 并发控制

- **文件锁**：暂未实现（MVP版本）
- **事务支持**：暂未实现（MVP版本）
- **备份恢复**：手动备份 JSON 文件

## 故障排查

### 问题：Load 命令无响应

**原因**：记忆数量过多，处理时间较长

**解决**：
1. 等待处理完成
2. 检查统计信息：`stats`
3. 必要时手动清理数据

### 问题：搜索结果不准确

**原因**：关键词匹配问题

**解决**：
1. 使用更精确的关键词
2. 尝试使用标签搜索
3. 检查记忆是否被正确记录

### 问题：存储空间过大

**原因**：原始记忆累积过多

**解决**：
1. 执行 `load` 生成总结
2. 删除旧的原始记忆文件
3. 只保留必要的原始记忆
